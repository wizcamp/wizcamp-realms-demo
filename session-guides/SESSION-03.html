üß† State vs Props

Before we dive into code, let‚Äôs understand the key difference between state and props ‚Äî two fundamental concepts that control how data flows in React apps.

Props vs State: The Key Differences

Props State
Data flows down from parent to child Data lives inside a component
Read-only ‚Äî child can‚Äôt change them Changeable ‚Äî component can update it
Like function parameters Like component memory
External data Internal data

Props are like ingredients you receive to make a recipe ‚Äî you can‚Äôt change them, but you use them to create something. State is like your kitchen‚Äôs current condition ‚Äî you can rearrange, add, or remove things as needed.

üí° State vs Props

Understanding this difference is crucial because it determines how data flows through your app and which component is responsible for managing what information. Props flow down from parent to child (one-way data flow), while state lives inside a component and can be updated by that component.

üìã Screen Constants

Before implementing navigation, let‚Äôs understand how constants organize your game‚Äôs screens and prevent errors.

File: src/constants/screens.js

Here‚Äôs the SCREENS object that defines your game‚Äôs navigation states:

export const SCREENS = {
  SPLASH: "splash",              // ‚Üê Initial screen users see
  PLAYING: "playing",            // ‚Üê Active gameplay screen
  GAME_OVER: "gameover",         // ‚Üê End screen after completing all zones
};

How Constants Work in Your Game

Constants are named values that stay the same throughout your app. Instead of typing the string "splash" every time you need to reference the splash screen, you use SCREENS.SPLASH.

When you write:

if (screen === SCREENS.SPLASH) {
  return <SplashScreen />;
}

JavaScript sees:

if (screen === "splash") {
  return <SplashScreen />;
}

The constant SCREENS.SPLASH resolves to the string "splash". This means you get autocomplete in your editor, protection from typos, and the ability to change the value in one place if needed.

In your game, you‚Äôll use these constants to:

  • Check which screen is currently active: screen === SCREENS.PLAYING
  • Change to a different screen: setScreen(SCREENS.GAME_OVER)
  • Render the right component: {screen === SCREENS.SPLASH && <SplashScreen />}

üí° Constants Create a Single Source of Truth

Constants create a ‚Äúsingle source of truth‚Äù ‚Äî one place where values are defined and referenced everywhere else. This same pattern works for any fixed values in your app: API endpoints, error messages, color themes, or game settings. When you see SCREENS.SPLASH in your code, you immediately know it‚Äôs a screen constant. Your code becomes self-documenting, making it easier to understand what‚Äôs happening at a glance.

üîó Context and Prop Drilling

Now that you‚Äôve seen the SCREENS constants, let‚Äôs understand why we use Context to share the current screen value across components.

The Prop Drilling Problem

Prop drilling is when you have to pass data through multiple component levels, even when the middle components don‚Äôt need that data. It‚Äôs like having to ask your friend to ask their friend to ask their friend for something ‚Äî inefficient and annoying.

Example of prop drilling:

App (has screen state)
 ‚Üì passes screen as prop
SplashScreen (doesn't need screen, just passes it along)
 ‚Üì passes screen as prop  
GameButton (finally uses screen)

Every component in the chain needs to accept and pass along the screen prop, even if it doesn‚Äôt use it. This creates brittle code that‚Äôs hard to maintain.

The Context Solution

Context lets any component access shared data directly without passing it through every level:

GameProvider (provides screen state)
 ‚Üì any component can access directly
GameButton (uses useGame hook to get screen)

With Context, components that need the screen value can grab it directly using the useGame hook. Components that don‚Äôt need it simply ignore it.

üí° Context Eliminates Unnecessary Passing

Context is React‚Äôs solution to prop drilling. It lets any component access shared data directly without passing it through every level. This keeps your code clean and makes it easy to add new components that need access to shared state. You‚Äôll use the useGame hook to access the current screen value from anywhere in your app ‚Äî no prop drilling required.

  // Add this line to access the shared state setter

  // ... rest of function ...

Define a function that changes the screen state to PLAYING when called, triggering the navigation to GameMap.

  const [showCredits, setShowCredits] = useState(false);
  const { setScreen } = useGame();
  
  // Add this function to handle game start
  const startGame = () => {
    setScreen(SCREENS.PLAYING);

  // ... rest of function ...

  text="Start Adventure"

Click the ‚ÄúStart Adventure‚Äù button on your splash screen.

‚úì You should see: The screen changes to GameMap! Your button now controls the entire app‚Äôs navigation through shared state.

üí° One Change Updates the Whole App

When you call setScreen(SCREENS.PLAYING), React updates the shared state in GameProvider and automatically re-renders all components that depend on that state. The App component sees the new screen value, evaluates its conditional rendering logic, and switches from <SplashScreen /> to <GameMap />. This is the power of centralized state management ‚Äî one function call orchestrates changes across your entire application.

üìã Essential Terms

Quick reference for all the state management concepts you just learned:

Term Definition Why it matters
üß† state Data that can change over time and causes components to re-render when it changes. State lets components ‚Äúremember‚Äù information and respond to user interactions dynamically.
ü™ù hook Functions starting with ‚Äúuse‚Äù that let you use React features like state and context. Hooks like useState are your tools for managing data and behavior in components.
üåê Context React‚Äôs solution to prop drilling ‚Äî lets components access shared data without passing props through multiple levels. Context prevents ‚Äúprop drilling‚Äù and provides shared state accessible from any component.
üîó prop drilling Passing data through multiple component levels, even when intermediate components don‚Äôt need that data. Context eliminates prop drilling by letting any component access shared data directly.
üîÑ useState A React hook that adds local state to functional components. useState gives individual components their own memory for data that only they need to track.
üìã constants Static values that don‚Äôt change, used to prevent typos and make code more maintainable. Constants like SCREENS.SPLASH prevent typos and make refactoring easier.
üîÄ conditional rendering Showing different components based on state or props using JavaScript expressions. Conditional rendering with && lets you control what users see based on app state.
üè™ Provider A Context component that makes shared state available to all child components. The Provider pattern wraps your app and gives all components access to shared data.
